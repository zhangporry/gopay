基于简历项目（**RocketMQ 消息总线重构、退款/充值链路实现、核心模块扩展性重构、QPS 提升 10%、消息丢失率 <0.01%**）整理的面试答题框架，覆盖高频追问点和深度连环炮。

## 一、项目介绍话术（STAR 法则）

### 标准版（1-2 分钟）

> 我负责的是公司支付系统的核心重构。原系统存在两个主要问题：一是**支付渠道与业务逻辑强耦合**，新增渠道需要改动大量代码；二是**回调处理采用同步大事务模式**，流量高峰时数据库连接被长时间占用，导致回调超时、消息丢失、客诉频发。
>
> 我的重构分三个方向：引入 **RocketMQ 实现回调异步削峰**，网关层只做验签解密后立即投递 MQ 并返回 200；基于**策略模式重构支付渠道接入层**，实现渠道热插拔；完整实现了**充值和退款业务链路**，包括状态机、幂等、对账。
>
> 最终效果：核心支付接口 **QPS 提升 10%**（长事务变短事务带来的连接复用红利），**消息丢失率从之前的偶发丢单降至 <0.01%**。

### 精简版（30 秒，二面/HR 面用）

> 我主导重构了公司的支付系统。核心改动是用 RocketMQ 把同步大事务拆成异步处理，同时用策略模式解耦了多支付渠道的接入。结果是 QPS 提升 10%，消息丢失率从偶发降到 0.01% 以下。

### 关键诱饵点

话术中故意埋了面试官大概率会追问的点：
- "异步削峰" → 追问 MQ 选型、可靠性保障
- "QPS 提升 10%" → 追问为什么引入中间件反而更快
- "消息丢失率 <0.01%" → 追问怎么做到的、怎么度量的
- "策略模式" → 追问设计模式的具体应用
- "状态机" → 追问资金安全如何保障
- "对账" → 追问长短款处理

## 二、架构演进：同步 → 异步（必考重点）

### 旧架构（同步大事务）

```text
微信/支付宝 → POST 回调
    → 网关开启数据库事务
    → 查单 + 悲观锁 (SELECT ... FOR UPDATE)
    → 更新订单状态
    → 调用充值/退款服务
    → 提交事务 (耗时 200ms+)
    → 响应 200 OK
```

**问题**：
1. 事务持有时间过长，DB 连接被大量占用
2. 微信回调超时阈值 5s，一旦并发上来，超时 → 微信判定失败 → 重试 → 更多请求 → 雪崩
3. 悲观锁导致锁竞争，同一订单并发回调时死锁风险

### 新架构（异步削峰）

```text
微信/支付宝 → POST 回调
    → 网关内存验签 + 解密 (~5ms)
    → 投递 RocketMQ (~10ms)
    → 立即响应 200 OK (总计 <20ms)

    → MQ 异步推送 → 消费者：校验 + 乐观锁更新状态 → 手动 ACK
```

**收益**：网关变成无状态的"消息搬运工"，连接持有时间从 200ms+ 降至 20ms，同样的连接池支撑了 10 倍以上的并发请求。

### 追问：为什么引入 MQ 后 QPS 反而提升了？

> 直觉上多了一跳网络开销应该更慢，但瓶颈不在单次请求延迟（latency），而在**并发吞吐量（throughput）**。
>
> 旧架构每个请求占用 DB 连接 200ms+，假设连接池 100 个连接，理论上限 = 100 / 0.2s = **500 QPS**。重构后网关层只需 20ms，理论上限 = 100 / 0.02s = **5000 QPS**。
>
> 10% 的 QPS 提升实际上是**长事务变短事务、连接快速复用**带来的宏观红利。同时 MQ 起到了削峰填谷的作用，下游消费者不会被瞬时流量打爆，整体系统稳定性也更好。

### 追问：为什么只提升了 10%，按你的计算应该提升很多倍？

> 理论值和实际值有差距，原因有三：
> 1. **瓶颈不只在回调链路**：下单、查单等其他接口也共享 DB 连接池，回调优化只释放了部分连接压力
> 2. **消费端吞吐量也有上限**：MQ 消费者更新 DB 时仍然需要连接，只是从"同步占用"变成了"异步错峰"
> 3. **10% 是真实压测数据**：我们没有虚报，而是在相同硬件条件下做 A/B 对比压测得出的结果
>
> 面试场景下这样回答很重要——**实事求是比吹牛更有说服力**。

## 三、消息可靠性：如何将丢失率压至 <0.01%（高频追问）

### 第一道防线：发端可靠投递

**问题**：网关向 MQ 发消息时，如果 MQ 宕机或网络抖动怎么办？

**方案**：本地消息表 + 定时补偿

```text
1. 回调到达 → 验签解密 → 写入本地 t_message_fallback 表（状态=待发送）
2. 尝试发送 MQ
3. 发送成功 → 更新 fallback 表状态为"已发送"
4. 发送失败 → 保留"待发送"状态
5. 定时任务（每 5s）扫描"待发送"且 next_retry < NOW() 的记录，重发
6. 重试采用指数退避：5s → 10s → 20s → 40s → ...
7. 超过最大重试次数 → 标记失败 → 告警人工介入
```

**追问：为什么不用 RocketMQ 的事务消息？**
> 事务消息适用于"本地事务和 MQ 消息需要原子性"的场景（比如"创建订单同时发 MQ"）。我们这里网关层本身没有本地业务事务要做，只是消息搬运，用本地消息表更轻量、实现更简单。事务消息还有半消息回查的复杂性，过度设计了。

**追问：本地消息表写入失败了怎么办？**
> 这时候网关还没响应微信 200 OK。微信发现超时或收到非 200 响应，会按策略重试（15s/15s/30s/3m/10m/20m/30m/30m/30m...共 24h）。所以网关写表失败 = 本次回调丢失，但**微信会重新通知**，下次再处理即可。这就是为什么微信有重试机制——它天然就是一道兜底。

### 第二道防线：收端幂等消费

**方案**：手动 ACK + 数据库乐观锁

```sql
-- 乐观锁更新：只有状态为 PAYING 时才更新为 SUCCESS
UPDATE t_pay_order SET status = 'SUCCESS', 
    transaction_id = ?, pay_time = NOW(), updated_at = NOW()
WHERE out_trade_no = ? AND status = 'PAYING';

-- affected_rows == 1 → 更新成功 → 提交 ACK
-- affected_rows == 0 → 已被处理 → 直接 ACK（幂等跳过）
```

- 消费者只在 DB 更新成功后才提交 ACK
- MQ 重复投递时，因为状态已经不是 PAYING，`affected_rows = 0`，天然幂等
- 不需要额外的 Redis 去重（减少依赖，DB 乐观锁本身就够用）

**追问：为什么用乐观锁而不是悲观锁？**
> 悲观锁（`SELECT ... FOR UPDATE`）在高并发时会导致锁等待和连接堆积，正是旧架构的痛点。乐观锁通过 WHERE 前置条件实现无锁并发控制，"失败"了直接跳过（因为说明已被处理），不需要重试。支付回调场景天然适合乐观锁——同一订单的状态更新是幂等操作，不存在"失败需要重试"的语义。

**追问：如果消费者更新 DB 成功，但 ACK 之前进程崩溃了怎么办？**
> MQ 没收到 ACK，会重新投递这条消息。消费者再次执行 UPDATE，因为状态已经是 SUCCESS，`affected_rows = 0`，直接 ACK。**乐观锁天然解决了这个问题，不需要额外处理。**

### 第三道防线：死信队列 + 主动查单

- 消费失败达到阈值（RocketMQ 默认 16 次重试，间隔从 10s 递增到 2h）→ 进入**死信队列 (DLQ)** → 触发告警，人工介入
- 系统额外有**定时主动查单任务**：扫描超过 N 分钟仍为 PAYING 状态的订单，调用微信/支付宝查单接口确认真实支付状态

```sql
-- 定时任务：每分钟扫描超时未完成的订单
SELECT out_trade_no, channel FROM t_pay_order
WHERE status = 'PAYING' AND created_at < DATE_SUB(NOW(), INTERVAL 5 MINUTE)
LIMIT 100;
-- 对每笔订单调用渠道查单接口 → 根据真实状态更新本地
```

### 追问：0.01% 的丢失率是怎么统计的？

> **T+1 日终对账**：
> 1. 每天凌晨下载微信/支付宝的前一天交易账单文件
> 2. 与系统内 SUCCESS 状态的订单做全量差异比对
> 3. **短款**（渠道有、我方无）= 消息丢失或消费失败 → 补单
> 4. **长款**（我方有、渠道无）= 可能伪造回调 → 风控审查
> 5. 丢失率 = 短款笔数 / 渠道总交易笔数
>
> 上线后连续监控一个月，日均交易 X 万笔，偶发 1-2 笔短款（主要来自极端网络分区），主动查单任务在 5 分钟内完成补偿。综合丢失率 < 0.01%。

## 四、核心模块重构：策略模式解耦

### 问题：原系统 if-else 地狱

```go
// 旧代码：每加一个渠道，改一堆 if-else
func Pay(channel string, order *Order) {
    if channel == "WECHAT" {
        // 微信支付逻辑 50 行...
    } else if channel == "ALIPAY" {
        // 支付宝逻辑 50 行...
    } else if channel == "QQ" {
        // QQ 支付逻辑 50 行...
    }
}
// 退款方法、查询方法、关单方法都要重复一遍这些 if-else
// 新增一个渠道 → 改 4 个方法 × 每个方法加一段 → 极易遗漏
```

**违反了开闭原则（OCP）**：每次扩展都要修改已有代码。

### 重构：策略模式 + 简单工厂

```go
// 1. 统一支付策略接口
type PaymentStrategy interface {
    CreateOrder(ctx context.Context, order *Order) (string, error)
    QueryOrder(ctx context.Context, orderNo string) (*OrderResult, error)
    Refund(ctx context.Context, refund *RefundReq) (*RefundResult, error)
    HandleNotify(ctx context.Context, req *http.Request) (*NotifyResult, error)
}

// 2. 微信实现（内部封装 gopay SDK 调用）
type WechatStrategy struct {
    client *wechat.ClientV3
    appId  string
}

func (w *WechatStrategy) CreateOrder(ctx context.Context, order *Order) (string, error) {
    bm := make(gopay.BodyMap)
    bm.Set("appid", w.appId).
        Set("description", order.Description).
        Set("out_trade_no", order.OrderNo).
        Set("notify_url", order.NotifyUrl).
        SetBodyMap("amount", func(b gopay.BodyMap) {
            b.Set("total", order.Amount) // 业务层已统一转为分
        })
    wxRsp, err := w.client.V3TransactionNative(ctx, bm)
    if err != nil {
        return "", err
    }
    return wxRsp.Response.CodeUrl, nil
}

// 3. 支付宝实现（注意金额单位差异）
type AlipayStrategy struct {
    client *alipay.ClientV3
}

func (a *AlipayStrategy) CreateOrder(ctx context.Context, order *Order) (string, error) {
    bm := make(gopay.BodyMap)
    bm.Set("subject", order.Description).
        Set("out_trade_no", order.OrderNo).
        Set("total_amount", fmt.Sprintf("%.2f", float64(order.Amount)/100)) // 分→元
    aliRsp, err := a.client.TradePay(ctx, bm)
    if err != nil {
        return "", err
    }
    return aliRsp.Response.TradeNo, nil
}

// 4. 工厂路由
func GetPaymentStrategy(channel string) PaymentStrategy {
    switch channel {
    case "WECHAT":
        return wechatStrategy  // 单例，初始化时创建
    case "ALIPAY":
        return alipayStrategy
    default:
        return nil
    }
}

// 5. 业务层调用（干净、统一、不关心具体渠道）
func HandlePay(ctx context.Context, channel string, order *Order) error {
    strategy := GetPaymentStrategy(channel)
    if strategy == nil {
        return fmt.Errorf("unsupported channel: %s", channel)
    }
    payUrl, err := strategy.CreateOrder(ctx, order)
    // ...
}
```

**收益**：新增渠道只需新增一个 struct 实现接口 + 在工厂 switch 中加一行，零修改已有代码。

### 追问：为什么不用 map 注册而用 switch？

> 渠道数量有限（<10 个），switch 可读性更好，编译期就能发现类型错误。如果渠道是动态配置的（比如运行时从数据库加载），才需要 map + 注册模式。**选型要根据实际场景，不要为了"高级"而高级。**

### 追问：如果面试官继续追问 Go 的 interface 和 Java 的有什么区别？

> Go 的 interface 是**隐式实现（鸭子类型）**——只要 struct 实现了接口的所有方法，就自动满足该接口，不需要显式声明 `implements`。这让代码更灵活、耦合度更低。
>
> Java 的 interface 是显式实现，编译期强制检查。Go 的方式更适合"面向行为"的设计，但也容易出现"意外满足接口"的问题，所以一般会加一行编译期检查：
> ```go
> var _ PaymentStrategy = (*WechatStrategy)(nil) // 编译期确认实现了接口
> ```

## 五、状态机设计：资金安全的核心保障

### 充值状态流转

```text
         创建订单          调起支付            回调/查单
  INIT ──────────> PAYING ──────────> SUCCESS
                     │                    
                     │ 支付失败/超时关单      
                     ├──────────> FAILED   
                     └──────────> CLOSED   
```

### 退款状态流转

```text
         发起退款              回调/查询
  (原订单 SUCCESS) ──> REFUNDING ──────────> REFUND_SUCCESS
                          │
                          └──────────> REFUND_FAILED
```

### 核心原则

1. **所有状态变更必须带前置条件**：`UPDATE ... WHERE status = '旧状态'`，从根源杜绝脏写
2. **禁止跳跃式状态变更**：不允许从 INIT 直接到 SUCCESS
3. **终态不可变**：SUCCESS/FAILED/CLOSED 等终态一旦写入不允许再修改
4. **退款独立建表**：退款不修改原支付订单的状态，而是新建退款单（一笔支付可能多次部分退款）

### 面试话术

> 没有严格状态机约束的资金操作就是裸奔。我所有涉及资金的 UPDATE 语句都带 `WHERE status` 前置条件，配合数据库行级锁，从数据层面保证不会出现状态错乱。比如一笔 PAYING 的订单，只有一个并发请求能成功将它更新为 SUCCESS，其余的 `affected_rows` 都是 0。

### 追问：PAYING 状态的订单如果用户关了页面怎么办？

> 两个机制兜底：
> 1. **订单超时关单**：创建订单时设置 expire_time（如 30 分钟），定时任务扫描超时订单，先调渠道查单 API 确认真实状态——如果渠道侧已支付成功就补单，如果未支付就调渠道关单接口 + 本地更新为 CLOSED
> 2. **渠道侧过期**：微信/支付宝的预支付单本身也有过期时间，过期后用户扫码会提示"订单已失效"

### 追问：如果用户支付成功了，但回调一直没到怎么办？

> 这就是主动查单任务的价值。定时扫描 PAYING 超过 5 分钟的订单，主动调渠道查单接口。如果渠道返回"已支付"，消费者补一次状态更新。**不依赖回调作为唯一的状态变更入口。**

## 六、数据库设计追问（面试高频）

### 追问：你的表结构是怎么设计的？

> 核心三张表：
> - `t_pay_order`：支付订单表，`out_trade_no` 唯一索引，status 字段做乐观锁
> - `t_refund_order`：退款单表，`out_refund_no` 唯一索引，与支付单通过 `out_trade_no` 关联
> - `t_message_fallback`：本地消息兜底表，保证 MQ 投递可靠性

### 追问：索引怎么设计的？

> - `out_trade_no` 上有唯一索引：保证订单号全局唯一 + 加速回调时的查单
> - `(status, created_at)` 联合索引：定时任务扫描超时 PAYING 订单用，避免全表扫描
> - `user_id` 索引：用户查询"我的订单"用
> - `t_message_fallback` 上 `(status, next_retry)` 联合索引：定时补偿任务扫描用

### 追问：订单量大了怎么办？分库分表方案？

> 1. **短期方案**：按时间分区（RANGE 分区），历史数据归档到冷库
> 2. **中期方案**：读写分离，查询走从库
> 3. **长期方案**：按 user_id 分片（ShardingSphere / TiDB），保证同一用户的订单在同一分片，方便查询
> 4. **分片后的挑战**：out_trade_no 查询需要额外处理——可以在订单号中编码分片键（如 `{userId后4位}_{时间戳}_{随机数}`），或者维护一张全局路由表

### 追问：为什么金额用 int 而不是 decimal？

> 1. int 运算不存在浮点精度问题（`0.1 + 0.2 != 0.3` 的经典坑）
> 2. int 比较和计算性能更好
> 3. 统一用"分"作为最小单位，业务层入参时转换，出参时按渠道格式化
> 4. 如果用 decimal，MySQL 的 DECIMAL(10,2) 也可以，但需要注意 ORM 层的映射和计算精度

## 七、Go 语言深度追问

### 追问：Go 的 context 在支付系统中怎么用的？

> 1. **超时控制**：每个 SDK 调用都传 `context.WithTimeout`，防止渠道接口卡死拖垮整个服务
> ```go
> ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
> defer cancel()
> wxRsp, err := client.V3TransactionNative(ctx, bm) // 超时自动取消
> ```
> 2. **链路追踪**：在 context 中传递 traceId，从网关到消费者全链路可追踪
> 3. **取消传播**：如果用户取消了支付页面，前端通知后端，后端通过 context.Cancel 取消正在进行的渠道调用

### 追问：goroutine 泄漏怎么排查？

> 1. `runtime.NumGoroutine()` 定期上报到监控，如果持续增长说明有泄漏
> 2. `pprof` 的 goroutine profile 可以看到每个 goroutine 的堆栈，找到阻塞在哪里
> 3. 常见泄漏场景：
>    - channel 没有消费者/生产者 → 永远阻塞
>    - HTTP 请求没设置超时 → 等待响应永远不回来
>    - context 没有 cancel → goroutine 永远不退出
> 4. 预防：所有对外请求必须带 timeout，所有 goroutine 必须有退出机制

### 追问：Go 的 GC 对支付服务有影响吗？怎么优化？

> 1. Go 的 GC 是并发标记清除，STW（Stop The World）时间通常在 1ms 以内，对支付服务影响很小
> 2. 但如果对象分配过多（比如每次请求都 new 大量临时对象），GC 压力会上升
> 3. 优化手段：
>    - `sync.Pool` 复用频繁创建的对象（如 BodyMap、buffer）
>    - 减少不必要的指针（指针多 → GC 扫描范围大）
>    - 设置 `GOGC` 环境变量调整 GC 频率（默认 100，调大则 GC 频率降低）
> 4. gopay 的 BodyMap 底层是 `map[string]any`，每次 make 会分配堆内存，高并发下可以考虑用 sync.Pool 复用

### 追问：你们服务是怎么做优雅退出的？

> ```go
> // 监听系统信号
> quit := make(chan os.Signal, 1)
> signal.Notify(quit, syscall.SIGINT, syscall.SIGTERM)
> <-quit
> 
> // 1. 先停止接收新请求（HTTP Server Shutdown）
> ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
> defer cancel()
> server.Shutdown(ctx) // 等待已有请求处理完毕
> 
> // 2. 等待 MQ 消费者处理完当前消息
> consumer.Shutdown()
> 
> // 3. 关闭 DB 连接池
> db.Close()
> ```
> 顺序很重要：先停止入口，再等处理完成，最后关闭资源。

## 八、安全相关追问

### 追问：支付回调怎么防止伪造？

> 1. **签名验证**：微信 V3 用 RSA 非对称签名，回调请求头带 `Wechatpay-Signature`，网关用微信平台公钥验签。没有私钥无法伪造签名。
> 2. **HTTPS**：回调地址必须是 HTTPS，防止中间人篡改
> 3. **IP 白名单**（可选）：部分渠道支持配置回调 IP 白名单
> 4. **gopay 的处理**：`wechat.V3ParseNotify()` 内部已封装签名验证逻辑，验签失败会返回 error

### 追问：如果有人抓包重放回调请求怎么办？

> 1. **签名中包含时间戳和随机数**：微信 V3 签名串包含 timestamp 和 nonce，时间戳过期（通常 5 分钟）的请求应该拒绝
> 2. **幂等保护**：即使重放成功通过了验签，乐观锁 `WHERE status='PAYING'` 保证不会重复处理
> 3. 所以核心防线还是**幂等**——即使签名验证被绕过，业务上也不会产生影响

### 追问：敏感信息（密钥、证书）怎么管理？

> 1. 禁止硬编码到代码中，禁止提交到 Git
> 2. 从配置中心（如 Nacos/Apollo）或密钥管理服务（如 Vault）动态加载
> 3. 部署时通过环境变量注入，容器内不落盘
> 4. 定期轮换密钥，微信平台证书通过 `AutoVerifySign()` 自动更新

### 追问：你们的监控是怎么做的？

> 核心监控指标：

| 指标 | 告警阈值 | 采集方式 |
| :--- | :--- | :--- |
| 支付成功率 | < 95% | 成功订单数 / 下单数 |
| 网关回调 P99 延迟 | > 100ms | Prometheus histogram |
| MQ 消费积压量 | > 1000 | RocketMQ Dashboard / 自定义 exporter |
| 退款成功率 | < 99% | 退款成功数 / 退款发起数 |
| 对账差异笔数 | > 0 | T+1 对账任务上报 |
| DLQ 消息数 | > 0 | RocketMQ DLQ topic 监控 |
| DB 连接池使用率 | > 80% | 连接池 metrics |
| goroutine 数量 | 持续增长 | runtime.NumGoroutine() |

### 追问：线上出了资金问题怎么排查？

> 1. 先通过 `out_trade_no` 查本地订单表，确认本地状态
> 2. 调渠道查单 API 确认渠道侧真实状态
> 3. 如果不一致 → 查 MQ 消费日志，确认消费是否成功
> 4. 查 t_message_fallback 表，确认消息是否投递成功
> 5. 查 DLQ，确认是否有消费失败的消息
> 6. **关键原则**：以渠道侧状态为准，本地状态可补偿修正

### 追问：如果凌晨 3 点告警说消费积压了怎么办？

> 1. **先看积压速度**：是瞬时尖峰还是持续积压？瞬时尖峰等 MQ 自动消化即可
> 2. **持续积压 → 看消费者日志**：是不是消费者挂了 / 抛异常 / DB 连接满了
> 3. **紧急扩容**：增加消费者实例数（RocketMQ Consumer Group 天然支持）
> 4. **最差情况**：消费者全挂 → 消息在 MQ 中堆积不会丢（RocketMQ 持久化） → 恢复后继续消费
> 5. **用户侧感知**：支付显示"处理中"，主动查单任务会在 5 分钟内兜底

## 十、高级追问 Q&A

### Q: 为什么选 RocketMQ 而不是 Kafka / RabbitMQ？

> - **vs Kafka**：Kafka 偏向大数据流处理，消息粒度是 partition 级别的顺序保证，延迟相对高（batch 攒批发送）。RocketMQ 原生支持事务消息、延迟消息、死信队列、消息重试，对支付场景更友好。
> - **vs RabbitMQ**：RabbitMQ 单机吞吐量有限（万级），且 Erlang 技术栈运维成本高。RocketMQ 是 Java 生态，团队更熟悉，且经过阿里双 11 验证，十万级 TPS。
> - 核心考量：**金融级可靠性 > 极致吞吐量**。RocketMQ 的同步刷盘 + 主从同步复制可以保证 broker 宕机不丢消息。

### Q: RocketMQ 消息顺序性怎么保证？

> 同一笔订单的消息（支付成功回调、退款回调）需要保证顺序消费。做法是：
> 1. 发送时用 `out_trade_no` 作为 MessageQueue 的选择 key，保证同一订单的消息进入同一个 queue
> 2. 消费者对每个 queue 单线程消费
> 3. 但其实在我们的场景中，乐观锁本身就保证了顺序——如果退款回调先到、支付回调后到，支付回调的 `WHERE status='PAYING'` 会命中成功，退款回调发现状态不对会重试，重试时支付已完成，就能正常处理了

### Q: 分布式事务怎么解决的？

> 支付系统不用强一致的分布式事务（2PC/TCC），用**最终一致性**：
> - 本地消息表保证"消息一定发出去"
> - MQ + 消费者保证"消息一定被处理"
> - 对账任务保证"最终状态一致"
> - 这是典型的 **BASE 理论**实践：基本可用（Basically Available）、软状态（Soft State）、最终一致（Eventually Consistent）
>
> **为什么不用 TCC？** 支付回调场景是单向的（渠道通知我们），不存在需要"回滚对方"的场景。TCC 适用于多方协调的场景（如同时扣库存+扣余额），我们的场景用最终一致性就够了，TCC 反而引入了不必要的复杂度。

### Q: 如何防止重复支付？

> 1. **前端防抖**：按钮点击后禁用，防止重复提交
> 2. **后端幂等**：`out_trade_no` 唯一索引保证同一订单不会重复创建；同一 `out_trade_no` 重复调用渠道下单接口，微信/支付宝会返回相同的预支付信息
> 3. **回调幂等**：乐观锁 `WHERE status='PAYING'` 保证只处理一次
> 4. **充值特殊处理**：充值场景下每次都要生成新的 `out_trade_no`，不能复用，否则金额不同会报错

### Q: 部分退款怎么实现？

> 1. 一笔支付订单可以发起多次退款，每次生成独立的 `out_refund_no`
> 2. 退款单表记录每次退款的金额，累计退款金额不能超过原订单金额
> 3. 发起退款前检查：`已退金额 + 本次退金额 <= 原订单金额`
> 4. 用分布式锁保证同一订单的退款请求串行执行，防止并发超退

### Q: 掉单了怎么处理？

> "掉单"是指用户实际支付成功了，但我们系统没有记录到。处理机制：
> 1. **主动查单**：定时任务扫描 PAYING 超过 5 分钟的订单，调渠道查单 API
> 2. **对账补偿**：T+1 对账发现短款，自动触发补单
> 3. **客服入口**：提供手动查单/补单的管理后台，客服可以通过渠道交易号手动触发状态同步
> 4. **用户自助**：用户在"我的订单"页面点击"支付遇到问题"，后端主动查单并刷新状态

## 十一、行为面试题

### Q: 这个项目中你遇到的最大挑战是什么？

> 最大的挑战是**保证重构过程中零资金事故**。支付系统不像普通业务可以灰度试错——金额少扣了用户投诉，金额多扣了是资损。
>
> 我的做法是：
> 1. **双写对比期**：新旧系统同时运行，新系统处理结果与旧系统对比，只有完全一致才切流
> 2. **灰度切流**：先切 1% 的流量到新系统，观察一周无异常后逐步放量
> 3. **快速回滚机制**：通过配置中心的开关，30 秒内可以切回旧系统
> 4. 最终平稳上线，零资金事故

### Q: 团队协作中有什么冲突吗？

> 在 MQ 选型时，有同事倾向于 Kafka（因为团队已有 Kafka 集群），我坚持用 RocketMQ。我的做法是：
> 1. 写了一份技术选型文档，从消息可靠性、延迟消息、死信队列、运维成本四个维度做了对比
> 2. 搭了一个 benchmark 环境，实测两者在"单消息可靠投递"场景下的表现
> 3. 最终用数据说服了团队
>
> 这件事让我意识到，**技术争论不要打嘴仗，用数据和事实说话。**

### Q: 如果让你重新做这个项目，你会有什么不同的做法？

> 1. **更早引入对账机制**：一开始我过于专注架构重构，对账是后面补的。实际上对账应该是第一天就有的，它是度量系统健康度的基础
> 2. **考虑引入 Go 的 errgroup 做并发控制**：早期消费者的并发控制写得比较朴素，后来发现 `golang.org/x/sync/errgroup` 更优雅
> 3. **日志结构化做得更早**：前期用的 xlog 打印文本日志，后面接入 ELK 时发现需要结构化日志，早做可以省很多工作

## 十三、面试节奏控制技巧

1. **主动出击**：不要等面试官一个个问，在回答中自然引出下一个话题。比如说完 MQ 削峰后主动加一句"为了保证 MQ 投递的可靠性，我们还设计了三道防线..."

2. **数字量化**：每个论点都配数据——"20ms"、"10%"、"0.01%"、"16 次重试"。没有数据的架构回答就是空谈。

3. **对比思维**：回答时用"旧系统怎么做 vs 我怎么做"的对比框架，面试官一听就明白你的价值。

4. **承认边界**：被问到不确定的问题，说"这个场景我在生产中没遇到过，但我的理解是..."，比胡编强 10 倍。

5. **反问环节准备**：
   - "贵公司的支付系统目前是什么架构？有哪些挑战？"
   - "团队对消息中间件的技术选型倾向是什么？"
   - "这个岗位日常会涉及资金安全或对账相关的工作吗？"
